# Review: 70-writing-readable-pipeline-scripts

---

## Issues found

### Critical

**1. Filename typo: "juman" → "human"**
The filename contains `juman readable` in two places. The title inside the file is correct
(`Writing Readable NEF-Pipelines Shell Scripts`). The filename needs renaming.

**2. Dangling reference: `convert_ligand_shifts.sh`**
Line 152 says "See `convert_ligand_shifts.sh` for a full working example." No `.sh` file
exists anywhere in the project. This is a broken reference that will confuse any reader or AI
following the skill. Either the file needs to be created and its path given, or the sentence
should be removed.

**COMMENT** added as an appendix

**3. Skills index not updated**
Both `skill - index of available expert workflow skills.md` and
`skills - index of available expert workflow skills.md` are auto-generated by
`scripts/generate_skill_index.py`. The new skill does not appear in either index, so it has
no `nef://skill/` URI and cannot be referenced by other skills or the MCP server.
`generate_skill_index.py` must be run after any rename/fix.

### Moderate

**4. Misleading Purpose section**
Lines 4–6 say:
> "The NEF-Pipelines MCP server allows the safe running of shell script pipelines."

This is inaccurate. The MCP server runs `nef_execute_pipeline` with a Python list of steps —
it does **not** execute bash scripts. The bash script conventions here are for producing
scripts that a **human** can take away and run outside the MCP server. The opening should
clarify this distinction, e.g.:
> "When a user asks for a pipeline expressed as a bash script they can run themselves,
> apply these conventions."

**5. CLI-Idioms reference style inconsistency**
Line 6 says "the CLI-Idioms resource" without the `nef://cli-idioms` URI used everywhere else
(`01`, `02`, `readme`, `skills` index all use the URI form). Should read:
> "…the `nef://cli-idioms` resource provides enough information to do this."

**6. No back-reference from skill 01**
`01-nef-pipelines-howto.md` has no mention of this skill. If a user asks an AI to "write me
a script", the AI following `01` won't know to read `70`. A one-liner in the
"When Helping Users" section of `01` would close the gap, e.g.:
> "User wants a bash script version of a pipeline → read `nef://skill/readable-pipeline-scripts`."

### Minor

**7. Convention 5 column-rename inconsistency**
The example inserts a column named `"Shift (ppm)"` (with a space), then renames it as
`"Shift_(ppm)=value"` (with an underscore). If NEF-Pipelines automatically converts spaces to
underscores in column names on insert, this should be noted explicitly. If it doesn't, the
rename key is wrong.

**COMMENT** The ad hoc tool kit does this when reading TSV/CSV files add is as a comment to the script and add a
note of it to the ad hoc skill document if its isn't present

**8. No `set -euo pipefail` explanation for zsh users**
The title and opening mention "bash/zsh" but `set -euo pipefail` behaves differently in zsh
(specifically `-o pipefail` has different semantics in older zsh). Worth a one-line note, or
drop "zsh" from the scope claim since all examples are explicitly `#!/usr/bin/env bash`.

**COMMENT** just go with bash

- Convention 4 (bare `\` as visual separator) is clearly explained with a working example.
- Convention 6 warning ("AIs are bad at calculating their positions") with the Python padding
  snippet is practical and honest.
- The mistakes table at the end is comprehensive and well-formatted.
- The canonical example is long enough to illustrate all conventions together.
- `nefl` used consistently throughout (correct for development/local use).
