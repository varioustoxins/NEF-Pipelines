# Circular Import Audit

Audit of imports annotated `# circular` in the codebase. Generated by static analysis
(AST import graph + transitive path check). See `check_circular.py` in job dir for the
script used.

---

These have a verified return path in the static import graph:

| `lib/nef_lib.py` | `from nef_pipelines.lib.namespace_lib import get_namespace` | `namespace_lib` → `nef_lib` |

## Caveats

- The script only traces static `import` and `from ... import` statements parsed by
  `ast`. Dynamic loading via `importlib.import_module()` or plugin registries is
  invisible to it.
- `nef_app_runner` uses dynamic module loading to register tools; edges from it to
  tool modules will not appear in the graph. This is why `mcp_lib → nef_app_runner`
  shows no return path even though it may be a real cycle at runtime.
- `nef_lib → structures`: `structures` is a low-level dataclass module with no
  upstream imports; the import in `nef_lib` is almost certainly just misplaced and
  should be moved to the top level.
